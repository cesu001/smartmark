## TODOs

- Build the PayPal payment/paywall feature, then revert `User.isPro` in `prisma/schema.prisma` to `@default(false)` and gate the Pro tier behind actual payment again (see `context/features/quick-fix02.md`).
- **Lighter note-list query (low priority).** The note list views are now paginated at 24/page, but each row still selects the full `content` column purely so `AppNoteCard` can render a 3-line preview. Deliberately not done in the 2026-07-18 batch: Prisma can't truncate a column in `select`, so it needs raw SQL plus a manual tag join, and the payload is already bounded. Worth revisiting only if very long notes make the list feel heavy.
- **Unbounded accumulation behind "Load more" (very low priority — largely defused).** `LoadMoreNotes` still only appends to its `notes` state, so cards accumulate without bound as you keep clicking. The expensive part is gone though: as of 2026-07-18 each card renders through `react-markdown` rather than its own Tiptap/ProseMirror editor, so what accumulates is a plain React/DOM tree instead of hundreds of live editor instances. Windowing/virtualizing would still be the fix if anyone ever reports sluggishness, but the original motivation for it is mostly resolved.
- **Note (not a bug):** a code-scanner pass on 2026-07-18 flagged the pagination cursor as missing an ownership check, on the theory that Prisma resolves the cursor anchor by id independently of the `where` filter. Tested directly and it's false: paginating the favorites set anchored on a non-favorite note that sorts earlier returned zero rows, where a filter-independent cursor would have returned rows. A foreign cursor matches nothing, exactly like a nonexistent one, so there's no probing primitive and no check is needed. Documented in `pageNotes`'s docblock so it isn't "fixed" later.
- **Note (not a bug):** code scanner flagged `NoteDrawer.tsx`'s Pin/Favorite active-state color (`text-primary/30`, fainter than the inactive `text-muted-foreground`) as "inverted" vs. `AppColCard.tsx`'s solid-color active-favorite convention. This was an explicit user design choice ("just change icon's color to primary/30"), not an oversight — left as-is, noted here in case it's revisited later.
