# Privacy Policy Research — Smark (smark.tw)

> **Purpose:** Groundwork for writing an accurate privacy policy for Smark. It maps
> what the app *actually* collects (from the code, not guesswork), lists every
> third-party that touches user data, identifies which privacy laws apply, and gives
> a section-by-section checklist for the policy document.
>
> **⚠️ Not legal advice.** This is engineering/product research to make the policy
> *accurate*. Before publishing, have the final text reviewed by someone qualified,
> especially the Taiwan PDPA and GDPR lawful-basis wording.
>
> _Compiled 2026-07-16. Verified against `prisma/schema.prisma`, `.env.example`,
> `src/lib/rate-limit.ts`, `src/components/AdSense.tsx`._

---

## 1. What Smark actually is (for the policy's "who we are" section)

- **Product:** AI-powered Markdown note-taking / knowledge base ("Smark").
- **Domain:** `smark.tw` (hosted on Vercel).
- **Audience:** Traditional-Chinese-speaking users (students, developers). Public
  marketing homepage + authenticated dashboard app.
- **Business model:** Free tier + Pro tier (Pro currently free, no payment processor
  wired up yet — see §8 gaps). Ads shown on **public pages only** via Google AdSense.

The operator's legal identity (individual vs. registered company, contact address,
and a contact email for privacy requests) must be filled in — the code doesn't
contain this. Taiwan PDPA and GDPR both require a named data controller + contact.

## 2. Personal data the app actually collects

Derived directly from the Prisma schema and auth/rate-limit code.

### 2.1 Account & identity data
| Data | Source | Where stored | Notes |
|---|---|---|---|
| Name (`User.name`) | User-entered (optional; register form has no name field, so often null) | Neon Postgres | Editable in profile |
| Email (`User.email`) | Registration / GitHub OAuth | Neon Postgres | Unique identifier |
| Password (`User.password`) | Registration | Neon Postgres | **bcrypt-hashed**, never plaintext |
| Avatar image (`User.image`, `User.avatarKey`) | Upload or GitHub avatar URL | Image bytes in **Cloudflare R2**; URL/key in Neon | |
| `emailVerified`, `isPro`, timestamps | System | Neon Postgres | |

### 2.2 OAuth (GitHub sign-in)
`Account` model stores OAuth tokens: `access_token`, `refresh_token`, `id_token`,
`scope`, `providerAccountId`, etc. Received from GitHub when a user chooses "Sign in
with GitHub."

### 2.3 User-generated content (the core product data)
| Data | Model | Notes |
|---|---|---|
| Note title + Markdown content | `Note` | Free-text; may contain whatever the user writes |
| **AI embeddings** of note content | `Note.embedding` (vector 1536) | Generated by sending note content to **OpenAI** |
| Collections (folders) | `Collection` | Names are user-entered |
| Tags | `Tag` | Names are user-entered |
| Favorite / pinned flags | Various | |

⚠️ **Sensitive-data caveat:** Notes are free-text. Users *can* paste anything —
health info, credentials, other people's personal data. The policy should state that
Smark is not intended for special-category/sensitive data and that users are
responsible for what they store. (This does not by itself create GDPR "special
category" processing, but it's worth a disclaimer.)

### 2.4 Session & security data
| Data | Where | Purpose |
|---|---|---|
| Session tokens (`Session.sessionToken`) | Neon + cookie | NextAuth session |
| Password reset tokens (`PasswordResetToken`) | Neon | 1-hour expiry, single-use |
| **IP address + email** | **Upstash Redis** (rate-limit keys) | Abuse/rate limiting on login, register, forgot/reset password, and AI endpoints. Login limiter keys on IP+email; others on IP. |

### 2.5 Automatically-collected / third-party-collected
- **IP address** — seen by Vercel (hosting), Upstash (rate limiting), and inherently
  by every request.
- **Ad-related data** — Google AdSense sets cookies / uses identifiers on **public
  pages** for ad serving & personalization (see §4 and §6).

## 3. Third parties / sub-processors (data recipients)

Every external service that receives user data. The policy must disclose these (both
PDPA and GDPR require naming categories of recipients; being specific builds trust).

| Service | Role | Data it receives | Location / transfer |
|---|---|---|---|
| **Vercel** | App hosting / CDN | All request data, IP, cookies | US-based; international transfer |
| **Neon** | PostgreSQL database (`shy-hill-60177324`) | All account + note data | Cloud region (AWS); international transfer |
| **OpenAI** | AI embeddings, note summaries, chatbot | **Note content** (title + body), chat messages | US-based; **AI processing of user content** |
| **Cloudflare R2** | Avatar image storage | Uploaded avatar images | International |
| **Resend** | Transactional email (password reset) | Recipient email address | Sends from `support@smark.tw` |
| **Upstash** | Redis rate limiting | IP address, email (as keys) | International |
| **GitHub** | OAuth login provider | Auth handshake; returns profile/email | US-based |
| **Google AdSense** | Advertising (public pages only) | IP, cookies, ad identifiers, browsing on public pages | Google; ad personalization |

> **OpenAI is the most important disclosure.** Users' note content and chatbot
> messages leave Smark and are sent to OpenAI for embedding/summarizing/chat. The
> policy must say this plainly. Note: OpenAI's API (not ChatGPT consumer) states it
> does **not** train on API data by default and retains it up to 30 days for abuse
> monitoring — worth stating, and worth verifying against OpenAI's current API data
> policy before publishing.

## 4. Cookies & tracking

| Cookie / storage | Set by | Purpose | Category |
|---|---|---|---|
| NextAuth session cookie | Smark | Keep users logged in | Strictly necessary |
| Google AdSense cookies | Google | Ad serving + personalization on **public pages** | Advertising / non-essential |

- **No first-party analytics** (no Google Analytics, PostHog, Sentry, Mixpanel, etc.)
  — confirmed by code search. Only AdSense.
- AdSense means the site uses **non-essential advertising cookies**, which triggers
  **consent requirements** under GDPR/ePrivacy for EU/EEA/UK visitors → a **cookie
  consent banner** is very likely required for those regions, plus Google's EU User
  Consent Policy obligations. This is currently **not implemented** (gap — see §8).
- AdSense is scoped to public pages only; the authenticated dashboard has no ads.
  The policy/cookie notice should reflect that distinction.

## 5. Which laws apply, and why

| Law | Applies because | Key implications |
|---|---|---|
| **Taiwan PDPA** (個人資料保護法) | `.tw` domain, Traditional-Chinese product, likely Taiwan-based operator & users | Notify data subjects at collection (purpose, categories, use period, rights, consequences of not providing); honor access/correction/deletion/stop-processing; consent for collection |
| **GDPR** (EU/EEA) | Global web service reachable by EU users; offering a service to them | Lawful basis per purpose; DSAR rights (access, rectification, erasure, portability, object, restrict); consent for ad cookies; international-transfer safeguards; breach notification; controller identity + (maybe) DPO/EU rep |
| **UK GDPR** | UK visitors | Mirrors GDPR |
| **CCPA / CPRA** (California) | If CA residents use it & thresholds met | "Do Not Sell/Share My Personal Information" — **AdSense personalized ads may count as a "sale"/"share"**; notice at collection; opt-out |
| **Google AdSense policies** | Contractual (publisher) | Must maintain a privacy policy disclosing cookies/third-party vendors & DoubleClick; obtain EEA/UK consent per Google's EU User Consent Policy |

**Note on thresholds:** CCPA/CPRA has revenue/volume thresholds a small project may
not meet, but the "Do Not Sell/Share" ad-cookie angle is the realistic trigger.
Taiwan PDPA and GDPR are the two that should drive the policy's structure.

## 6. Required content — privacy policy section checklist

A single policy can satisfy all frameworks if it covers:

1. **Who we are** — operator legal name, `smark.tw`, contact email for privacy
   requests, (physical/registered address for PDPA).
2. **What data we collect** — the §2 categories: account, OAuth, note content,
   embeddings, session/security, IP, ad data.
3. **How we collect it** — directly from the user, via GitHub OAuth, automatically
   (IP/cookies), generated (embeddings).
4. **Why / purposes + legal basis** —
   - Provide the service (contract): account, notes, collections, tags.
   - AI features (contract / consent): sending note content to OpenAI for
     embeddings, summaries, chatbot.
   - Security & abuse prevention (legitimate interest): rate limiting via IP/email.
   - Email delivery (contract): password reset via Resend.
   - Advertising (consent, esp. EEA/UK): AdSense on public pages.
5. **Who we share with** — the §3 sub-processor table; state we don't sell data
   except to the extent ad personalization is a "sale/share" under CCPA.
6. **International transfers** — data goes to US/global providers (Vercel, Neon,
   OpenAI, etc.); name the safeguard (e.g. SCCs / provider DPAs).
7. **Cookies** — necessary session cookie vs. AdSense advertising cookies; link a
   cookie section or separate cookie policy; consent mechanism for EEA/UK.
8. **Retention** — how long: account data until deletion; reset tokens 1h;
   rate-limit keys short-lived (Redis window); note content until user deletes or
   closes account. **Define concrete periods** (currently undefined — gap §8).
9. **User rights** — access, correction, deletion, portability (export already
   exists via `.md` export!), object/restrict, withdraw consent, complain to a
   supervisory authority. Explain how to exercise (email + in-app: profile edit,
   note delete, **Delete Account** already cascades all data).
10. **Security** — bcrypt password hashing, HTTPS, access controls, rate limiting.
11. **AI-specific disclosure** — content sent to OpenAI; not used to train models
    (verify); how embeddings work.
12. **Children** — state minimum age; not directed at children under 13/16.
13. **Changes to this policy** — how updates are communicated; "last updated" date.
14. **Contact / complaints** — privacy contact; EU supervisory authority & Taiwan
    authority reference.

## 7. Features that *help* compliance (already built — cite them in the policy)

- **Account deletion** — `DELETE /api/dashboard/user` cascade-deletes all notes,
  collections, tags, sessions, and the R2 avatar → supports GDPR erasure / PDPA
  deletion rights. Say this explicitly; it's a strong point.
- **Data export** — notes export to `.md` (single/bulk) → supports data portability.
- **Profile edit** — name/avatar/password self-service → supports rectification.
- **Password hashing** — bcrypt; passwords never stored or logged in plaintext.
- **Enumeration-safe password reset** + rate limiting → security posture to mention.

## 8. Gaps & decisions to resolve BEFORE publishing

These are places where the code doesn't answer the question and a human must decide:

1. **Operator legal identity & contact** — individual or company? What email/address
   for privacy requests? (Required by PDPA + GDPR.)
2. **Retention periods** — currently no defined retention for account/note data
   beyond "until deleted." Pick concrete periods (e.g. inactive-account cleanup?).
3. **Cookie consent for EEA/UK** — AdSense = non-essential cookies. There is
   **no consent banner implemented**. Either (a) add a consent mechanism (Google's
   Consent Management Platform / CMP requirement for EEA/UK traffic), or (b) decide
   to serve only non-personalized ads / restrict audience. This is a real compliance
   gap, not just wording.
4. **CCPA "Do Not Sell/Share"** — if targeting/serving California users with
   personalized ads, may need an opt-out link.
5. **OpenAI data handling claim** — verify OpenAI's *current* API data-retention &
   no-training-by-default terms before asserting them in the policy.
6. **Age limit** — decide and state (13? 16? aligns with GDPR consent age, which
   varies 13–16 by member state).
7. **Payment data** — none collected yet (Pro is free, no PayPal). Once PayPal is
   added, the policy must be updated to cover payment processing (PayPal as
   processor; Smark likely never sees card data).
8. **Governing law / jurisdiction** — likely Taiwan; confirm.
9. **`ads.txt` is live** (`public/ads.txt`) — confirms the AdSense relationship is
   active, reinforcing that the ad-cookie disclosure is mandatory, not hypothetical.
